## 一、项目概述

### 1.1 项目背景

钓迹App是一款面向钓鱼爱好者的社区类产品，当前首页鱼情指数依赖随机数生成，数据可信度低；排行榜功能缺乏地域层级划分，鱼种维度数据缺失；附近功能仅有粗糙的地图展示，无法有效帮助用户发现周边作钓地点。本次重构旨在提升首页数据可信度、信息丰富度和用户决策效率。

### 1.2 改版目标

| 目标维度 | 具体描述 |
|---------|---------|
| **数据可信** | 鱼情指数由真实数据驱动，数值可解释、可追溯 |
| **决策效率** | 用户进入首页即可获得"去哪钓"的初步决策依据 |
| **地域价值** | 排行榜支持省/市/区三级穿透，鱼种排行补充维度 |
| **社交激励** | 前三名突出展示，激发用户竞争和活跃动机 |

### 1.3 技术约束

- **前端：** Vue3 + Vant4（移动端H5）
- **后端：** Spring Boot + MySQL
- **数据更新频率：** 鱼情指数 T+1 日结（初期）；排行榜实时计算

### 功能 1：钓点热度榜（替换原鱼情指数）

#### 2.1.1 设计变更（2026-04-20）

原设计「鱼情指数」因数据基础不足（上钩率字段缺失、用户评分表为空）暂时搁置。替换为「钓点热度榜」，直接统计真实上报次数，数据更可信、实现更简单。

#### 2.1.2 用户价值

热度榜直观展示"哪些钓点最近最活跃"，用户一看就知道哪里人多、哪里有鱼。相比抽象的评分数字，上报次数更真实、更容易建立信任。热度高的钓点往往饵料充足、钓法验证过，值得优先尝试。

**用户价值评分：⭐⭐⭐⭐⭐（核心功能，P0）**

#### 2.1.3 数据来源与计算逻辑

- **数据表：** `t_fish_record`（渔获记录表）
- **热度值：** 近30天 / 本月，该钓点的渔获记录条数
- **过滤条件：** 仅统计 `deleted=0` 的有效记录
- **排序规则：** 按热度值降序，取 TOP 100

**SQL 示例：**
```sql
-- 近30天热度榜
SELECT s.id, s.name, s.city,
       COUNT(r.id) AS heat_score
FROM t_fishing_spot s
LEFT JOIN t_fish_record r ON r.spot_id = s.id
  AND r.created_at >= DATE_SUB(CURDATE(), INTERVAL 30 DAY)
  AND r.deleted = 0
GROUP BY s.id
HAVING heat_score > 0
ORDER BY heat_score DESC
LIMIT 10;
```

#### 2.1.4 页面呈现

- **位置：** 首页顶部第一模块，渐变深蓝背景卡片
- **时间切换：** 「近30天」「本月」Tab 切换（默认显示近30天）
- **展示数量：** TOP 3（完整排行跳转排行榜页）
- **名次样式：** 🥇🥈🥉 金银铜渐变底色 + 白色数字
- **无数据：** "暂无数据，成为第一个分享者？"

#### 2.1.5 MVP范围

✅ **MVP实现：** 近30天热度 TOP 3，本月切换，数据来自 t_fish_record

### 功能 2：排行榜（省/市/区三级 + 前三名突出）

#### 2.2.1 用户价值

排行榜提供横向比较视角，满足用户的竞争心理和社交炫耀需求。省/市/区三级穿透让用户既能看到全国高手，又能找到同城同区的竞争对手，降低竞争门槛，提升"我也能上榜"的参与感。

**用户价值评分：⭐⭐⭐⭐（核心功能，P0）**

#### 2.2.2 地域层级设计

| 层级 | 入口 | 排行维度 |
|-----|------|---------|
| **全国榜** | 默认Tab | 总渔获重量 / 总作钓次数 |
| **省级榜** | Tab切换 → 选择省份 | 同上 |
| **市级榜** | Tab切换 → 选择省份 → 选择城市 | 同上 |
| **区级榜** | Tab切换 → 选择省→市→区 | 同上 |

> 地域选择器采用级联下拉组件（Vant Cascader），支持搜索快速定位。

#### 2.2.3 前三名样式突出

| 排名 | 样式 | 说明 |
|-----|------|------|
| 🥇 第1名 | 金色头像框 + 皇冠图标 + 动画光效 | 居中大号展示 |
| 🥈 第2名 | 银色头像框 + 排名徽章 | 次居中 |
| 🥉 第3名 | 铜色头像框 + 排名徽章 | 次居中 |
| 4–10名 | 数字排名 + 头像 + 昵称 + 战绩 | 普通列表 |
| 10名后 | 分页加载 | 需触发加载更多 |

#### 2.2.4 数据来源与计算逻辑

- **数据表：** `fishing_record`（钓行记录）、`user`（用户基础信息）
- **排行维度（可切换）：**
  - 总渔获重量（kg）：SUM(record.weight)
  - 总作钓次数：COUNT(record.id)
  - 总渔获条数：SUM(record.count)
- **过滤条件：** 仅统计近90天内的有效记录
- **计算频率：** 每日凌晨批量计算 + 实时增量更新
- **榜单数量：** 每榜展示TOP 100

#### 2.2.5 MVP范围

✅ **MVP实现：** 全国榜 + 省级榜（默认开放省份），支持"总渔获重量"单一维度，前三名突出样式。城市榜和区级榜、鱼种维度、鱼种筛选在迭代版本实现。

### 功能 3：鱼种排行（按鱼种维度展示）

#### 2.3.1 用户价值

鱼种排行从"目标鱼"视角切入，帮助用户快速找到"近期什么鱼好钓"——这是钓友们最常问的核心问题之一。相比用户排行，鱼种排行提供的是**钓点×鱼种**的交叉信息，对用户的决策指导更直接。

**用户价值评分：⭐⭐⭐⭐（增强功能，P1）**

#### 2.3.2 页面结构

入口：**排行榜 Tab** 内嵌子Tab「鱼种榜」

```
鱼种榜
├── 鱼种选择器（下拉，支持搜索，如"鲫鱼""翘嘴""鲤鱼"）
├── 鱼种总览列表（按该鱼种总捕获量排序，默认展示TOP 10钓点）
└── 详情入口（点击钓点进入钓点详情页）
```

**列表字段：**

| 字段 | 说明 |
|-----|------|
| 排名 | 1–10 |
| 钓点名称 | 地理名称 |
| 总捕获量(kg) | 该鱼种在该钓点的累计捕获量 |
| 最近捕获时间 | 最近一条该鱼种记录的日期 |
| 上钩率 | 该鱼种在该钓点的上钩比例 |

#### 2.3.3 数据来源与计算逻辑

- **数据表：** `fishing_record`（含 `fish_species` 字段）
- **过滤条件：** 近90天有效记录
- **计算频率：** 每日批量计算
- **鱼种数据：** 需提前维护鱼种字典表（`fish_species_dict`），建议录入国内常见淡水鱼30–50种

#### 2.3.4 MVP范围

✅ **MVP实现：** 默认展示"综合热门鱼种"（跨鱼种汇总TOP钓点），单鱼种精确排行在迭代版本实现。

### 功能 4：附近地图（市级标点 + 距离排序）

#### 2.4.1 用户价值

钓鱼是强地域属性的活动，用户天然关心"我家附近哪里能钓"。市级地图标点解决了"只知道区级范围，但不了解市级钓点"的问题，距离排序让用户快速锁定最便利的选择。

**用户价值评分：⭐⭐⭐⭐（增强功能，P1）**

#### 2.4.2 页面结构

```
附近地图模块
├── 地图视图（默认展示，以用户当前位置为中心）
│   └── 市级钓点标点（圆形图标，颜色=鱼情指数色）
│       └── 点击标点 → 气泡卡片（钓点名、鱼情指数、距离）
└── 列表视图（切换Tab，显示距离排序的钓点清单）
    └── 按距离由近到远排列，显示钓点名 + 距离(km) + 鱼情指数
```

**市级说明：** 地图标点以"地级市"为单位聚合，同一城市内多个钓点以_cluster聚合显示，点击展开。

#### 2.4.3 数据来源与排序逻辑

- **位置数据：** 用户GPS坐标（需申请定位权限） + 钓点经纬度
- **距离计算：** Haversine公式（两点间球面距离），单位km，保留1位小数
- **排序逻辑：** 默认按距离升序；支持切换"按鱼情指数排序"
- **展示范围：** 默认展示半径50km内钓点；支持手动调整半径（10km / 25km / 50km / 100km）
- **数据来源：** `fishing_spot`（钓点表，含经纬度字段）

#### 2.4.4 权限与体验

- **首次进入：** 弹窗申请定位权限，用户拒绝时展示默认城市（基于IP粗略定位）
- **无数据提示：** 该范围内无钓点时显示"附近暂无数据，成为第一个分享者？"

#### 2.4.5 MVP范围

✅ **MVP实现：** 列表视图（距离排序），地图视图在迭代版本实现（地图SDK接入成本较高）。

## 四、数据模型设计（概要）

### 4.1 核心数据表

```sql
-- 钓行记录表
CREATE TABLE fishing_record (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    spot_id BIGINT NOT NULL,
    fish_species VARCHAR(50),          -- 鱼种（关联字典表）
    weight DECIMAL(10,2),               -- 捕获重量(kg)
    catch_count INT DEFAULT 0,         -- 捕获条数
    hook_count INT DEFAULT 0,          -- 上钩次数（用于计算上钩率）
    has_valid_catch TINYINT(1),        -- 是否有有效上钩(0/1)
    record_date DATETIME,
    latitude DECIMAL(10,6),
    longitude DECIMAL(10,6),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 钓点表
CREATE TABLE fishing_spot (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    province VARCHAR(50),
    city VARCHAR(50),                   -- 地级市（用于附近功能）
    district VARCHAR(50),
    latitude DECIMAL(10,6) NOT NULL,
    longitude DECIMAL(10,6) NOT NULL,
    spot_type ENUM('river','lake','reservoir','sea'),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 用户表（扩展字段）
CREATE TABLE user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    nickname VARCHAR(50),
    avatar_url VARCHAR(255),
    province VARCHAR(50),
    city VARCHAR(50),
    district VARCHAR(50),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 鱼种字典表
CREATE TABLE fish_species_dict (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,         -- 如"鲫鱼""翘嘴"
    category VARCHAR(20),               -- 如"淡水鱼"
    is_active TINYINT(1) DEFAULT 1
);
```

### 4.2 排行榜计算SQL（每日定时任务）

```sql
-- 每日凌晨计算：省级渔获量排行（近90天）
SELECT
    u.id AS user_id,
    u.nickname,
    u.avatar_url,
    u.province,
    SUM(r.weight) AS total_weight,
    COUNT(r.id) AS total_trips
FROM fishing_record r
JOIN user u ON r.user_id = u.id
WHERE r.record_date >= DATE_SUB(CURDATE(), INTERVAL 90 DAY)
  AND u.province = :province
GROUP BY u.id
ORDER BY total_weight DESC
LIMIT 100;
```

## 七、风险与依赖

| 风险 | 描述 | 应对方案 |
|-----|------|---------|
| 数据不足 | 初期钓行记录少，鱼情指数可信度低 | 上线引导弹窗鼓励用户上报渔获；无数据钓点显示"数据积累中" |
| 数据质量 | 用户上报数据可能存在虚报 | 引入举报机制；设置单用户日上报上限；异常数据清洗 |
| 定位精度 | GPS定位误差导致距离排序偏差 | 拒绝精确定位时使用IP粗略定位；列表展示"约XXkm" |
| 性能 | 排行榜实时计算大表JOIN | 定时任务预计算+缓存（Redis），接口读缓存 |
| 设计师排期 | 页面布局样式待设计师输出 | 产品先输出结构初稿，设计师可基于此评审 |
